# Lazyweb MCP Research Workflow

この手順書は、Lazyweb MCPを使ってUI/LP改善または記事ヘッダー画像の参考調査を行うための標準ワークフローです。既存の `docs/lazyweb-reference-policy.md` を前提に、参照元をコピーせず、抽象パターンだけを記録します。

## 基本ルール

- Lazyweb MCPは参考調査と抽象パターン抽出にだけ使う。
- ロゴ、人物写真、独自UI、独自イラスト、ブランド固有の配色や構図は再現しない。
- スクショ原本、参照画像、画像URL、署名付きURL、認証情報、個人用設定値、MCP接続設定の実体は保存しない。
- 調査ログには、調査日、目的、入力条件、短い要約、抽象パターン、使わない要素、最終案への反映だけを残す。
- 最終成果物は参照元と混同されないオリジナル表現にする。

## 1. 調査前に決めること

調査を始める前に、次の項目を短く書く。

- 調査日:
- 調査目的:
- 対象:
- 成功条件:
- 使うルート: UI/LP改善 または 記事ヘッダー画像

UI/LP改善では、対象画面とconversion goalを明確にする。記事ヘッダー画像では、記事の主題とヘッダーで出したい感情を明確にする。

## 2. Lazyweb MCPで取得する

主にモバイルA/Bテスト evidence 検索を使う。入力条件は、後から再現できる程度の要約だけをログに残す。

記録する入力条件:

- category:
- productまたはtarget context:
- conversion goal:
- constraints:
- visual inspection budget:
- high design bar指定:

Lazyweb MCPがタイムアウトする、利用できない、または対象に合う evidence が少ない場合は、通常Web調査にフォールバックする。その場合も `docs/lazyweb-reference-policy.md` に従い、URL、取得日、媒体種別、抽象パターンだけを残す。

## 3. UI/LP改善ルート

UI/LP改善では、画面を直接真似するのではなく、A/B差分から改善仮説だけを抽出する。

記録するもの:

- 対象画面:
- conversion goal:
- evidenceの短い要約:
- 参考になったA/B差分:
- 抽出した抽象パターン:
- 実装に反映する仮説:
- 採用しない要素:

抽象化の観点:

- ファーストビューの情報階層
- CTAの目立たせ方
- 比較、安心材料、社会的証明の置き方
- 入力前後の摩擦を減らす構成
- 画面内の情報密度と余白

使わないもの:

- 実在アプリの画面構成そのもの
- ブランド固有の色、文言、アイコン
- 実験画面のスクショ、画像、UIコピー
- 参照元のcontrol/variantを再現する実装

## 4. 記事ヘッダー画像ルート

記事ヘッダー画像では、ビジュアルを直接真似せず、構図、余白、色、情報密度だけを抽出する。

記録するもの:

- 対象記事:
- 主題:
- ヘッダーで出したい感情:
- evidenceの短い要約:
- 抽出した抽象パターン:
- 使わない要素:
- オリジナル案への反映:

抽象化の観点:

- 主役モチーフの位置
- タイトル安全領域としての余白
- 色数とコントラスト
- 静物、カード、メモ、生活道具などの大まかなモチーフ
- 画面全体の情報密度

使わないもの:

- 実在メディアのロゴ、記事タイトル、ナビ、広告
- 実在人物写真、商品パッケージ、店舗名、価格
- 参照元に近いレイアウト、配色、写真構図
- 読める文字や実在サービスのUI

## 5. ログ作成

調査後は `docs/lazyweb-reference-log-template.md` を複製して、用途に合う欄だけを埋める。不要なルートは削除してよい。

ログに残すevidenceは要約だけにする。画像、スクショ、署名付きリンク、認証情報、個人設定は残さない。

## 6. Safety Review

ログの最後に必ずSafety Reviewを置き、次を確認する。

- 参照元をコピーしていない。
- 抽象パターンのみ利用している。
- スクショ原本や参照画像をGit管理していない。
- ロゴ、人物写真、独自UI、独自イラストを真似していない。
- 認証情報、個人用設定値、MCP接続設定の実体を書いていない。
- 最終成果物が参照元と混同されない。

## 7. 最終確認

作業後に次を確認する。

- Markdownの見出し、表、チェック項目が崩れていない。
- 保存禁止情報をログ本文に残していない。
- UI/LP改善では、仮説が対象画面とconversion goalに対応している。
- 記事ヘッダー画像では、構図、余白、色、情報密度が抽象化されている。
- Lazyweb MCPが使えなかった場合のフォールバック理由が書かれている。
